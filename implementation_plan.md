# 东方酒艺 (Touhou Brews) 品质系统重构与产品机制设计

基于你的需求，我们将对核心产品机制进行彻底的底层重构，引入 **NBT驱动的品质星级系统 (0-5星)**、基于流程准确度动态变化的品质，以及一套占位符视觉机制。以下是具体的实施计划：

## User Review Required

> [!WARNING]
> **这涉及对当前机器数据流的破坏性重构！** 
> 引入“配料比”和“处理过长过短”会改变原本“配方吻合即全自动走读条并输出成品”的死板逻辑。我们需要将机器改造为“**玩家主导的动态炼成系统**”。请仔细阅读下方的设计提议并给出意见。

---

## Proposed Changes

### 1. 通用酒系统引擎 (NBT Item 架构)

所有产出的酒类将不再是死板的单例物品，而是统一重构为继承自自定义 `BrewItem` 的子类。该类读取物品的 NBT `Quality` 标签（0-5 整数）。

* **动态本地化命名 (Dynamic Naming):**
  * **0 星 (Quality=0)**: 彻底失败，重写本地化键，名称强制变为“失败的酒渣”或“残缺液体”。没有任何正面药水效果。
  * **1-4 星 (Quality=1,2,3,4)**: 一般品质，名称将会降级为“普通清酒”、“乡村梅酒”或“廉价葡萄酒”，显示对应的星级，药水效果数值下降或消失。
  * **5 星 (Quality=5)**: 完美品质，解锁传说名称（例如：“蕾米莉亚的血红葡萄酒”），全额赋予对应的高阶药水效果。
* **贴图占位替换:** 
  替换所有的成品酒 `.json` Item Model，一律套用 `minecraft:item/potion` 及其 overlay。通过 Fabric Client `ColorProviderRegistry` 为不同的子类注入占位特定颜色（如红酒=深红，梅酒=青绿）。

### 2. 机器控制变量 (酿造品质算法)

为了实现“原料过多/过少”、“时间过长/过短”导致降级，需要将配方判定的逻辑重写：
* **比例影响:**
  机器允许非完美比例的原料输入（比如本来需要 `1个酒醪+1个水瓶`，你放了 `3个酒醪`）。比例偏离完美比例越多，**品质基础上限** 就越低。
* **时间惩罚机制 (以发酵桶和压榨床为例):**
  机器不再“自动将成品排入第 4 槽并停止”。
  改为：当开始工作时，内部形成“发酵液”并推进进度。进度从 `0% -> 100% -> 200%`。
  * `100%` 时品质最佳。
  * 未满或过载都会导致品质扣除。
  玩家必须通过右键机器（或特定按键）**主动提取**成品，提取的瞬间计算总分（基础分 - 比例扣分 - 时长偏离扣分），生成对应 NBT 星级的饮料给你。

### 3. 创造模式管理方块 (Admin Brew Block)

* **[NEW] `AdminBrewBlock` & `AdminBrewBlockEntity`**: 
  新增一款只存在于创造模式物品栏的特殊机器“神之酿酒台”。
* **GUI / 交互**:
  右击打开一个简易界面，或者直接左键轮换类型、右击轮换星级。玩家可以直接点击该方块来凭空生成一张带有指定 NBT `{Quality: X}`，并指定种类（清酒/红酒/梅酒）的饮品用于测试。

## Open Questions

> [!IMPORTANT]
> 这几个设计决策需要你拍板确认：

1. **提取机制:** 如果玩家必须在“刚好达到 100%”时提取才能获得 5 星，这意味着机器**不能用漏斗自动抽取成品**，否则漏斗可能会在 0% 或 100% 时瞬间把东西抽走。你希望为了品质机制彻底**禁用自动化（漏斗/传输管道）**，还是给机器设定一种“停止键”？
2. **“废品”处理:** 0 星出来的产物，你是希望它就是一个名为“残废的酒渣”的纯垃圾物品（没有用只能扔掉），还是允许玩家把它当做劣质食物/水喝掉（带剧毒/反胃）？
3. **原料过多过少:** 原本 UI 里原料堆叠直接放一组 `64` 个进去，机器会一个个自动扣。如果我们要计算“原料过多过少”，你是否倾向于机器“开始”时直接清空当前槽位中**所有**物品，也就是你放多少个进去，机器就吃多少个来计算这一批酒的配比？
